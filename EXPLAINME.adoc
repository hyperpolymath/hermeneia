# EXPLAINME: Hermeneia

> The README makes claims. This file backs them up.

Hermeneia is an experimental voking / imminence query language for Vocarium.

It asks what stored particulars can become under a declared use-model. It is not SQL with mythological names. It is a language for calling trope-shaped records into use, relation, challenge, crossing, comparison, or withdrawal.

## 1. What the README claims

The README should claim only this much:

> Hermeneia is an experimental query language for voking stored particulars from Vocarium under declared use-models.

It should not claim:

> Hermeneia solves interpretation.

or:

> Hermeneia automatically knows what things mean.

or:

> Hermeneia replaces human judgement.

Hermeneia makes interpretive calls explicit enough to inspect, check, and contest.

---

## 2. Core claim: Hermeneia is a voking language

### README claim

Hermeneia does not merely select rows. It vokes tropes.

### What this means

To voke is to call a stored particular into a use-relation.

The root is useful because it points to a family of operations:

```text
invoke    — call into use
evoke     — call forth resonance
convoke   — call together
transvoke — call across context or use
provoke   — call into challenge
intervoke — call between paths
revoke    — call back / withdraw licence
```

Hermeneia’s query act is therefore not just:

```text
find matching records
```

but:

```text
call this particular into this use and show whether it can answer here
```

### Receipts to look for

A mature repository should define the core voke operations in a grammar or language reference:

```text
docs/syntax.md
docs/voke-operations.md
grammar/hermeneia.ebnf
examples/invoke.hil
examples/provoke.hil
examples/transvoke.hil
```

### Caveat

If the current implementation only has one search command, the README should say:

> Hermeneia is designed as a voking language.

not:

> Hermeneia implements the full voke family.

## 3. Core claim: Hermeneia is an imminence language

### README claim

Hermeneia asks what a particular is about to become under a use.

### What this means

A database record is not automatically an answer. A stored trope becomes answer-like only when brought under:

```text
a purpose
a context
a use-model floor
a warrant
a requested response role
```

Hermeneia queries the moment before answerhood.

It asks:

```text
Can this trope become evidence?
Can this paraphrase become a submitted claim?
Can this quotation become an interpretation?
Can this AI-generated sentence become evidence of student understanding?
```

This is what “imminence” means here: not mystical futurity, but the threshold at which a stored particular is about to count as something for a use.

### Receipts to look for

Hermeneia should eventually produce outputs that include not just records, but response roles:

```text
can become:
  evidence
  paraphrase
  loose gloss
  candidate claim
  warrant
  failed substitution
```

### Caveat

Do not overclaim. Hermeneia does not make the future happen. It checks whether a particular can be licensed as a response under a declared use-model.

## 4. Core claim: Hermeneia queries use, not only data

### README claim

Hermeneia queries particulars under use-models.

### What this means

The same trope may pass under one use-model and fail under another.

Example:

```text
"real language"
```

may be sufficient as:

```text
a loose study note
```

but insufficient as:

```text
a technical paraphrase of "authentic language"
```

Hermeneia must therefore make the use-model explicit.

Example shape:

```text
invoke "real language"
against "authentic language"
under use_model "critical-paraphrase"
show verdict, loss, warrant
```

### Receipts to look for

There should be examples where changing the use-model changes the verdict.

Minimum example set:

```text
examples/use-models/casual-note.json
examples/use-models/critical-paraphrase.json
examples/results/same-candidate-different-verdicts.json
```

### Caveat

If the query language does not yet support explicit use-models, the core claim is still design intent, not implementation evidence.

## 5. Core claim: Hermeneia separates retrieval from licence

### README claim

Hermeneia distinguishes:

```text
retrieved
interpretable
licensed
```

### What this means

A normal query result can encourage false confidence. Hermeneia should refuse that collapse.

A candidate can be:

```text
retrieved: yes
interpretable: yes
licensed: no
```

This is central to the system.

Example:

```text
"croaked" may be retrieved as a lexical neighbour of "passed away".

It may be interpretable as death-related.

It is not licensed for a funeral-condolence use-model.
```

### Receipts to look for

Hermeneia output examples should show:

```text
verdict
loss-shape
failed floor
reason
missing warrant
```

not just a ranked list of results.

### Caveat

If Hermeneia only returns matches, it has not yet earned the claim that it is a licensing-aware query language.

## 6. Core claim: Hermeneia has a family of voke operations

### README claim

Hermeneia’s operations are organised around modes of call.

### Canonical operations

#### invoke

Call a particular into a declared use.

```text
invoke "authentic language"
under use_model "critical-paraphrase"
show verdict, loss, warrant
```

#### evoke

Call forth neighbouring or latent particulars.

```text
evoke around "authentic language"
within context "de Man / rhetoric"
limit quality_loss <= Attenuated(3)
```

#### convoke

Call several particulars together.

```text
convoke [
  "authentic language",
  "poetry",
  "myth",
  "madness"
]
under use_model "critical-reading"
show constellation
```

#### transvoke

Call a particular across context, bearer, record, or use.

```text
transvoke "authentic language"
from context "source quotation"
to context "student paraphrase"
show path, loss, failed_floors
```

#### provoke

Call a candidate into challenge.

```text
provoke "real language"
against "authentic language"
under use_model "technical-paraphrase"
show failure_points
```

#### intervoke

Call between trope paths or candidates.

```text
intervoke path "quote-to-paraphrase"
with path "quote-to-summary"
show divergence
```

#### revoke

Call back or withdraw a previous licence.

```text
revoke verdict "v_2041"
because warrant "source_check_failed"
```

### Caveat

These operations may begin as design vocabulary before they are all implemented. The README should mark the implemented subset clearly.

## 7. Relationship to Vocarium

### README claim

Hermeneia queries Vocarium.

### What this means

Vocarium stores:

```text
tropes
paths
grades
warrants
use-models
verdicts
```

Hermeneia calls those records into use.

Hermeneia should not be responsible for long-term storage. Vocarium should not be responsible for interpreting every query. The boundary matters.

```text
Vocarium = storage of vokeable particulars
Hermeneia = query acts over those particulars
```

### Receipts to look for

There should eventually be a fixture where:

```text
Vocarium record → Hermeneia query → Hermeneia result
```

is visible end to end.

## 8. Relationship to Haec

### README claim

Haec transforms; Hermeneia queries.

### What this means

Haec records transformation paths such as:

```text
attenuate[2] u
project[{quality, record}] v
fuse[source_quote] x y
```

Hermeneia asks whether those transformed particulars can be used.

Example:

```text
Haec:
  "authentic language" → "real language"
  grade: quality = Attenuated(6)

Hermeneia:
  invoke transformed trope
  under use_model "critical-paraphrase"
  verdict: p-insufficient
```

### Caveat

Hermeneia should not duplicate Haec’s transformation effects as query verbs. Haec effects and Hermeneia vokes are different layers.

```text
Haec effects:
  preserve / project / collapse / detach / attenuate / fuse

Hermeneia vokes:
  invoke / evoke / convoke / transvoke / provoke / intervoke / revoke
```

## 9. Evidence boundary

Hermeneia may currently be one of several things:

```text
conceptual language design
grammar draft
parser prototype
query planner
Vocarium-integrated query runner
checker-integrated language
```

The README should say which one is true.

Do not claim:

```text
production query engine
complete hermeneutic calculus
automated interpretation
verified query semantics
```

unless the repository contains direct evidence.

The honest public framing is:

> Hermeneia is an experimental voking query language for asking what stored particulars can become under declared use-models.

## 10. What Hermeneia is not

Hermeneia is not:

```text
SQL with poetic verbs
a general-purpose search engine
a thesaurus
a semantic oracle
a truth engine
a replacement for human interpretation
a proof that an answer is good
```

Hermeneia is a language for making the interpretive call explicit and inspectable.

It can expose:

```text
loss
warrant
floor
verdict
missing evidence
failed role
```

It cannot remove the need for judgement.

## 11. Falsifiability and failure modes

Hermeneia’s claims fail if:

1. Queries do not require or expose use-models.
2. Results are only ranked matches, not verdict-bearing candidates.
3. The same candidate cannot pass under one use-model and fail under another.
4. The language cannot show what was lost.
5. Voke operations collapse into vague synonyms for search.
6. `provoke` cannot produce useful failure maps.
7. `revoke` cannot withdraw earlier licences.
8. The system hides O2 obligations instead of surfacing them.
9. Haec effects and Hermeneia vokes are mixed into one confused layer.

A useful hard test:

```text
"passed away" → "croaked"
under use_model "funeral-condolence"
```

Hermeneia should not merely retrieve both as death-related expressions. It should show why the candidate is not licensed under that use.

## 12. Next receipts needed

Recommended next evidence items:

```text
1. A minimal Hermeneia grammar.
2. A voke operation table with implemented / planned status.
3. One invoke example that checks a Vocarium record against a use-model.
4. One provoke example that shows why a candidate fails.
5. One transvoke example that traces a particular across contexts.
6. One revoke example that withdraws a verdict after warrant failure.
7. JSON result shape:
   query
   candidates
   loss-shape
   floor
   verdict
   reasons
   missing warrants
8. A design note separating:
   retrieved
   interpretable
   licensed
9. A no-overclaiming status table:
   Designed / Parsed / Planned / Implemented / Tested / Integrated.
```

Until those receipts exist, the safe claim is:

> Hermeneia is a proposed voking query language for trope-aware databases.

Once they exist, the stronger claim becomes:

> Hermeneia is an implemented query layer for calling Vocarium records into use-modelled interpretation, challenge, crossing, and revocation.
